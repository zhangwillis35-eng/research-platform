Phase 0 — 产品愿景

为什么要做 ScholarFlow？它要解决什么问题？

User我需要一个平台，覆盖管理学研究的完整流程：从文献检索、综述撰写、知识图谱构建、研究想法生成、理论整合到最终的研究提案——每一步都有 AI 辅助，但研究者保持主导权。
Claude核心设计原则确定：AI 作为「研究副驾驶」而非自动驾驶——每个模块提供 AI 分析结果，但研究者可以审阅、修改、选择性采纳。采用 Next.js App Router 全栈架构，DeepSeek V4 Flash 作为默认 AI 引擎（成本低、速度快），所有 LLM 调用使用 SSE 流式传输实时展示结果。

核心决策：LLM 系统 prompt 统一使用英文（提高推理质量），输出统一中文（面向中文用户）。所有检索关键词必须翻译为英文（中文仅用于 CNKI）。

① 文献检索 — 多源聚合与 AI 评分

设计与实现

smart-search.ts · 1200+ 行5 个数据源
User需要一个智能检索系统，能从多个学术数据库同时搜索，自动翻译中文关键词，按相关性和期刊等级排序。
Claude设计多源检索管道：用户输入 → LLM 提取关键词 + 同义词扩展 + 生成精确/宽泛双模式查询 → 并行检索 Google Scholar / Semantic Scholar / OpenAlex / arXiv / CORE → 去重（DOI + 外部 ID + 标题模糊匹配）→ 期刊元数据富化（JCR、IF、SSCI/SCI）→ AI 相关性评分（1-10）→ 分级排序输出。

AI 评分维度：关键匹配度、学术贡献、方法论创新、数据来源质量。每篇论文附带评分理由。

期刊标识：FT50（50 刊）、UTD24（24 刊）、ABS 4*（46 刊）徽章实时显示，经 CEIBS LibGuides / UT Dallas / AJG 2024 逐一校验。

Google ScholarSemantic Scholar OpenAlexarXivCORE SSE StreamingAI Scoring

检索管道优化战（跨 3 个会话反复调试）

平台最频繁调试的组件
Bug 1: 评分上限失控

200+ 篇论文全部做 AI 评分，极慢。添加预截断：按期刊等级排序后取 limit×3 再评分。但这又导致小众高质量论文（CHI/CSCW）被丢弃，最终设计「质量源优先」策略：ABS 3+、JCR Q1、arXiv 优先保留。

Bug 2: 期刊过滤 AND → OR

用户发现 80 篇论文过滤后仅剩 1 篇。根因：targetJournals 和 requireUTD24 是 AND 条件。改为 OR 逻辑。

Bug 3: 覆盖率不足

用户指出 Energy Economics 和 TFSC 期刊的论文搜不到。OpenAlex 单次仅取 30 条 → 扩展到 200 条。

Bug 4: 结果前后不一致

LLM 关键词提取 temperature=0.2 引入随机性 → 降为 0。排序添加 title-based tie-breaking。

Bug 5: 双重过滤导致 0 结果

filterByRelevance(score≥3) + applyTieredLimit(score≥5) 双重过滤。评分 3-4 的论文通过第一关被第二关杀掉。移除预过滤。

反复出现的模式：扩大覆盖 → 超时/限流 → 缩小覆盖 → 漏论文 → 再扩大。最终平衡点：OpenAlex 200条 + 质量源优先排序 + enrichment cap = limit+20。

Phase 4 — Bug 修复与性能优化

从级联 Bug 到系统性能危机——每一次故障如何推动架构进化

性能危机：85 个 Hooks 导致浏览器卡死

系统级 Bug
User打开网页还是会卡，Mac 经常鼠标失灵
Root Cause85 个 usePersistedState hooks 在组件挂载时同时从 sessionStorage 读取和解析 JSON，阻塞主线程数百毫秒
Fix将 sessionStorage 读取包裹在 requestIdleCallback 中，延迟到浏览器空闲时执行。结合 prefetch={false} 减少页面预加载

Phase 6 — 协作模式复盘

40 天 · 4 个会话 · 477+ 轮交互中浮现的人机协作规律

User 的角色

  • 产品定义者 — 统一身份、两栏布局、三引擎等方向性决策
  • 质量哨兵 — 在真实场景中发现 Bug（检索不全、评分不一致、鼠标卡死）
  • 策略指引者 — 集成理论知识库、移植想法模块等高层策略
  • 体验守护者 — 检索结果有序、速度要快、对话要有记忆

Claude 的角色

  • 架构实现者 — 数据模型、API 路由、前端组件、SSE 管道
  • 根因诊断者 — AND/OR 逻辑、双重过滤、字段不匹配、85 hooks 并发
  • 知识工程师 — 35+ OB 理论库、期刊列表官方校验
  • 性能调优者 — 并发控制、token 裁剪、idle callback、预截断

反复出现的模式

  • 覆盖率 vs 速度拉锯 — 检索管道跨 3 个会话反复调试，扩大 → 超时 → 缩小 → 漏论文 → 再扩大
  • 级联 Bug — 修一个 Bug 引入另一个（预截断 → 丢弃小众论文 → 0 结果）
  • 字段名不一致 — 管道和前端之间的命名不匹配，反复出现
  • 缓存数据与新代码不兼容 — sessionStorage 旧数据导致新页面崩溃

关键架构决策

  • 三引擎并存 — 从用户追问自然演化，非预设方案
  • GROBID 替代 unpdf — Docker 本地的 ML 解析器，质量远超规则解析
  • 理论知识库分层筛选 — 全量用于分析，按 topic 裁剪版用于快速生成
  • SSE 统一流式架构 — 检索/综述/想法/提案/翻译全部使用 SSE 实时推送
  • 跨功能数据流 — 领域空白 → 研究想法 → 研究提案，一键传递
